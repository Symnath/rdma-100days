## 📚 知识点

1. Why DCQCN？

- RoCEv2 承载层是 UDP，无 TCP 滑动窗口、内置重传逻辑；
- RoCE 依赖 PFC 实现链路无损，但 PFC 只解决单跳链路不丢包，无法解决多流并发涌入交换机缓存导致的**队列拥塞**；
- 单纯 PFC 会引发 PFC 死锁、全局队头阻塞，必须上层 DCQCN 做端到端流控，主动降速发送端流量。

DCQCN：**Data Center Quantized Congestion Notification**

数据中心量化拥塞通知，是 RoCEv2 配套的**硬件端到端拥塞控制算法**，和二层 PFC 无损流控配合，形成完整无损网络，解决 AI 训练 Incast 拥塞、时延抖动问题；区别于 TCP 软件拥塞算法，DCQCN 无 CPU 开销、响应更快。


### DCQCN 核心工作流程（三段式）

#### 1. 交换机检测拥塞 & 生成 CNP 拥塞通知包

交换机端口队列占用超过预设阈值时，对 RoCE 报文标记 ECN（显式拥塞标记）；

接收网卡收到带 ECN 标记的报文，封装**CNP（Congestion Notification Packet）** 反向发给发送方 QP。

#### 2. 发送端网卡硬件收到 CNP，调整发送速率

网卡硬件维护每个 QP 的速率状态机，收到 CNP 后：

- 快速降低发送速率（量化衰减）；
- 无 CNP 持续一段时间则缓慢提升发送速率；
    
    全程**硬件执行，不经过 CPU / 内核**，无软件开销。

### 3. 恢复逻辑（量化递增）

持续无 CNP 反馈时，以小步长逐步提升发包速率，平衡吞吐与低时延。



2. IB/RoCEv2/iWARP

| 对比维度       | IB (InfiniBand)             | RoCEv2                                     | iWARP                             |
| ---------- | --------------------------- | ------------------------------------------ | --------------------------------- |
| **承载网络**   | IB 专用交换机 / 线缆，无法复用现有以太网     | 标准以太网交换机（25G/100G/200G）                    | 任意标准 TCP 以太网，普通交换机即可              |
| **可靠传输机制** | IB 硬件原生可靠，内置信用流控            | BTH 硬件重传，依赖二层 PFC 防丢包                      | 复用 TCP 滑动窗口、重传，无硬件无损依赖            |
| **跨三层路由**  | 原生不支持三层，仅子网内通信              | 支持 IP 三层路由（UDP 4791 端口）                    | 原生支持三层 TCP 路由，跨机房无限制              |
| **拥塞控制**   | IB CC                       | DCQCN（网卡硬件卸载）                              | TCP CUBIC/reno，软件拥塞               |
| **链路无损依赖** | 硬件原生无损，无需 PFC               | 必须开启 802.1Qbb PFC，否则大规模 Incast 极易丢包        | 完全不需要 PFC，丢包由 TCP 重传兜底            |
| **部署成本**   | 极高，全套 IB 交换机、专用光模块溢价大       | 中等，现有万兆 / 100G 以太网改造即可，仅需交换机支持 PFC         | 最低，复用普通商用以太网，无特殊交换机要求             |
| **时延表现**   | 最低（亚微秒级）                    | 次低（微秒级，无损网络下接近 IB）                         | 更高，TCP 协议栈带来额外时延抖动                |
| **CPU 开销** | 极低，硬件全卸载                    | 极低，BTH 硬件处理，内核旁路                           | 偏高，TCP 协议栈 CPU 消耗大                |
| **运维复杂度**  | 极高，独立运维栈，和以太网割裂             | 中等，需调优 PFC/DCQCN，避免 PFC 死锁                 | 极低，运维逻辑和传统 TCP 网络完全一致             |
| **典型业务场景** | 超大规模超算、HPC、高端 AI 训练集群       | 国内大模型分布式训练、公有云 AI 算力集群（西云 AWS、NVIDIA 生态主流） | 金融低延迟交易、传统云数据库、跨地域远距离 RDMA        |
| **厂商生态**   | Mellanox/NVIDIA IB 交换机，封闭生态 | NVIDIA ConnectX、Spectrum 交换机，AI 算力行业事实标准   | Intel E800、部分 Broadcom 网卡，金融行业使用多 |

## 🎯 今日总结
### Q1：排查 RoCE 静默丢包，除 PFC/DCQCN 外最易忽略的两个配置点是什么？

A：优先核对全链路 MTU 是否统一为 9000，以及两端网卡的 802.1p 优先级映射是否一致。

### Q2：RoCEv2 跨三层组网时，DCQCN 的 CNP 拥塞通知包如何路由回发送端？

A：CNP 封装为标准 UDP/IP 报文，依托普通 IP 路由原路返回，无需额外专用转发配置。

### Q3：PFC 死锁应急恢复后，用什么快速验证网络已恢复正常？

A：用 ib_write_bw 打满带宽，确认网卡无丢包计数、端口 Pause 帧不再增长即可。
